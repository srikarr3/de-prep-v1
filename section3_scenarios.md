## Scenario 1: Design an end-to-end pipeline for an e-commerce company needing daily sales reports
* **The Interviewer's Question:** *"We need to generate daily sales reports by 8 AM for our business team. The data comes from our transactional PostgreSQL database and clickstream events on our website. How would you design this end-to-end?"*
* **What they are testing:** 
  * Ability to handle heterogeneous data sources (transactional DB vs streaming clickstream).
  * Selection of orchestration tools and storage layout.
  * Practical understanding of Medallion architecture and data ingestion rates.
* **Structured Step-by-Step Approach:**
  1. **Ingestion Layer:** Use Debezium to stream CDC logs from the PostgreSQL transactional DB into Kafka. For clickstream, use a lightweight API gateway to write events directly to a Kafka topic.
  2. **Storage Layer:** Mount Azure ADLS Gen2 or AWS S3 as the Data Lake. Use Databricks to write Kafka topics into Delta Lake **Bronze tables** in an append-only format.
  3. **Transformation Layer (Silver):** Run a scheduled PySpark job (orchestrated by Airflow or Databricks Workflows) at 4:00 AM. Read the Bronze Delta tables, cast data types, handle nulls, deduplicate based on `order_id`, and write to **Silver tables** using `MERGE` statements.
  4. **Aggregation Layer (Gold):** Run a downstream PySpark job to calculate daily sales metrics (revenue per category, active store locations). Write results to a **Gold Delta table** or load them into a Data Warehouse (e.g., Snowflake) using its Spark connector.
  5. **Orchestration & SLA:** Schedule the pipeline in Apache Airflow. Add an SLA check of 2 hours. Implement Slack notifications to alert on failures.
* **Common Mistakes:**
  * Suggesting querying the production transactional database directly for reports (knocks out production DB performance).
  * Skipping the deduplication step (creates double-counting in sales reports).
  * Proposing a streaming pipeline for the final report when the business only reviews it daily (over-engineering, increases cloud costs).

## Scenario 2: Your streaming pipeline is receiving late-arriving events — how do you handle them?
* **The Interviewer's Question:** *"In our streaming ingestion pipeline, some user actions occurring on mobile apps are received hours late because of network disconnection. How do you handle these late-arriving events without corrupting daily aggregates?"*
* **What they are testing:** 
  * Understanding of Event Time vs Processing Time.
  * Knowledge of Watermarking and State Management in streaming engines like Spark/Flink.
  * Design strategies for updating historical records.
* **Structured Step-by-Step Approach:**
  1. **Define Event Time:** Ensure the pipeline uses the event's creation timestamp (`event_time`) generated by the mobile client, rather than the ingestion timestamp (`processing_time`) of the broker.
  2. **Implement Watermarking:** Configure a watermark threshold (e.g., `withWatermark("event_time", "2 hours")`). This instructs the streaming engine to maintain state for up to 2 hours of delay to aggregate events.
  3. **Configure Windowed Aggregations:** Apply windowing (e.g., `.groupBy(window(col("event_time"), "1 hour"))`). Late events arriving within the 2-hour window will update the active state store.
  4. **Handle Out-of-Watermark Data (Dead Letter / Late Sink):** Events older than the 2-hour watermark threshold are discarded by the streaming window. Write these discarded events to an "audit/late-data" Delta location for manual backfilling.
  5. **Apply Merges:** Write downstream streaming updates using Delta Lake's `MERGE` operation to update historical partitions without corrupting current ones.
* **Common Mistakes:**
  * Suggesting using Processing Time (which incorrectly groups late data into the current hour's aggregates).
  * Proposing keeping an infinite watermark window (which causes memory leaks on worker nodes by keeping state stores active forever).
  * Not writing late events to a fallback location (leads to data loss).

## Scenario 3: A pipeline fails midway through writing 10M records — how do you ensure no data loss or duplication?
* **The Interviewer's Question:** *"Your daily ETL pipeline is writing 10 million rows to a database. The cluster crashes when it has written 6 million records. How do you design your pipeline to handle this failure cleanly?"*
* **What they are testing:** 
  * Understanding of transactional writes and idempotency.
  * Knowledge of two-phase commits, temporary staging locations, and Delta Lake's ACID transaction log.
* **Structured Step-by-Step Approach:**
  1. **Write to Staging Directory first:** Ingest and transform data, then write the complete output to a temporary staging folder in the data lake (e.g., `s3://bucket/staging/`).
  2. **Use Delta Lake Table (Target):** Use Delta tables as target storage. Delta Lake's transaction log provides ACID guarantees. When Spark writes, the change is committed in a single step in the transaction log (`_delta_log/xxxx.json`). If the cluster crashes, the transaction is not committed, and readers see only the pre-job state.
  3. **Implement Idempotent Merge/Overwrite:** If writing to a non-ACID database, use a `MERGE` statement based on a unique business key (e.g., `transaction_id`), or truncate the target partition and perform a full overwrite on retry.
  4. **Use Kafka Offset Commits (if streaming):** Commit Kafka offsets *only after* the database write transaction has been successfully confirmed.
* **Common Mistakes:**
  * Using simple `INSERT` or `.write.mode("append")` directly to the target table without a transaction framework (causes duplicate records on retry).
  * Suggesting deleting the table and restarting the pipeline without considering downtime for downstream readers.
  * Committing Kafka offsets before confirming database writes.

## Scenario 4: How do you optimize a Spark job that's running slow?
* **The Interviewer's Question:** *"A nightly Spark job that normally takes 20 minutes is now taking 3 hours to complete. Walk me through how you would diagnose and fix this bottleneck."*
* **What they are testing:** 
  * Debugging skills using the Spark UI.
  * Understanding of Spark performance bottlenecks: data skew, disk spill, partitioning, and network shuffles.
* **Structured Step-by-Step Approach:**
  1. **Analyze Spark UI:** 
     * Look at the **SQL Tab** to see the physical execution plan and locate long-running stages.
     * Check the **Stages Tab** and look at the Task metric distribution (Min, 25th, Median, 75th, Max). If Max task duration is 2 hours while Median is 5 seconds, you have **Data Skew**.
     * Look for **Spill (Memory)** and **Spill (Disk)** indicators.
  2. **Fix Data Skew:** 
     * If the skew is caused by a join key, use salting (adding random suffixes to keys to distribute them across executors).
     * If joining a large and a small table, force a `broadcast()` join on the small table.
  3. **Optimize Partitioning:** Ensure partition sizes are in the 128MB–200MB range. Adjust `spark.sql.shuffle.partitions` (default is 200) based on your data volume.
  4. **Enable Predicate Pushdown:** Verify that filter conditions are pushed down to the storage layer, minimizing the amount of data read into memory.
* **Common Mistakes:**
  * Immediately increasing cluster size / upgrading instance types (increases cloud costs without fixing the underlying software bottleneck).
  * Randomly changing Spark configuration parameters without analyzing metrics in the Spark UI.
  * Forgetting that data skew is a data layout problem that cannot be solved by simply adding more workers.

## Scenario 5: How do you handle schema changes in production without breaking downstream consumers?
* **The Interviewer's Question:** *"The backend team changes a column data type from INT to STRING and deletes an unused column in a MySQL table. How do you handle this schema migration in your downstream analytics pipeline without breaking client dashboards?"*
* **What they are testing:** 
  * Schema management, registry governance, and data contracts.
  * Medallion layer isolation (Bronze vs Silver vs Gold).
* **Structured Step-by-Step Approach:**
  1. **Establish a Data Contract:** Define schemas and check compatibility rules using a central Schema Registry (e.g., Confluent Schema Registry). Reject backward-incompatible changes at the CI/CD pipeline level.
  2. **Isolate in Bronze Layer:** Use Schema Evolution (`.option("mergeSchema", "true")`) in the Bronze/raw layer so the new string column and deleted columns do not crash raw data ingestion.
  3. **Enforce Schema in Silver Layer:** In the Silver transformation layer, enforce the expected target schema. Cast the incoming column back to the target format (e.g., using `coalesce` or typecasting `cast("int")`) and add default null values for missing columns.
  4. **Gold Views / API layer:** Present data to downstream users via SQL Views rather than raw tables. This allows you to alias or modify columns underneath without changing the interface seen by downstream dashboards.
* **Common Mistakes:**
  * Letting schema changes propagate directly to Gold tables, which crashes BI tools.
  * Overwriting historical integer records with string values, causing serialization errors.
  * Not having a data contract or communication channel between backend engineers and the data team.

## Scenario 6: Design a pipeline that ingests CDC data from a MySQL database into a Data Lakehouse
* **The Interviewer's Question:** *"We need to replicate transactional database changes from MySQL to our Delta Lakehouse in near real-time. How would you design this?"*
* **What they are testing:** 
  * Knowledge of Change Data Capture (CDC) mechanisms.
  * Integration of Kafka, Debezium, and Delta Lake.
  * Management of updates and deletes in a data lake.
* **Structured Step-by-Step Approach:**
  1. **Log Extraction:** Enable binary logging (`binlog`) on the MySQL database. Configure **Debezium** as a connector to read binlog events asynchronously.
  2. **Streaming Pipeline:** Stream the CDC change events into a Kafka topic (one topic per database table). Payload should contain the before/after state of each row, along with operation metadata (Insert, Update, Delete).
  3. **Ingestion (Bronze):** Run Spark Structured Streaming to read from Kafka and append the raw CDC events directly to a Delta Lake **Bronze table** (append-only log).
  4. **Reconstruction (Silver):** Run a scheduled or continuous Spark Streaming job with a `foreachBatch` writer. Execute a `MERGE` statement against the target **Silver Delta table** to apply updates, inserts, and delete operations based on the source metadata key.
* **Common Mistakes:**
  * Querying MySQL tables with timestamp columns (e.g., `WHERE updated_at > last_run`) at high frequencies, which slows down the transactional database.
  * Neglecting to handle delete operations (leads to orphan records in the data lakehouse).
  * Running `MERGE` directly from Kafka on every event (bottlenecks memory; grouping into micro-batches first is critical).

## Scenario 7: How do you implement SCD Type 2 in a Delta Lake pipeline?
* **The Interviewer's Question:** *"We need to track historical changes in our customer addresses. How would you implement a Slowly Changing Dimension (SCD) Type 2 table using PySpark and Delta Lake?"*
* **What they are testing:** 
  * Analytical understanding of SCD Type 2 logic.
  * Mastery of Delta Lake's `MERGE` syntax and schema structures.
* **Structured Step-by-Step Approach:**
  1. **Define Schema:** The target dimension table must have attributes like `customer_id`, `address`, `effective_date`, `end_date`, and `is_current` (boolean).
  2. **Identify Incoming Changes:** Read source updates and compare them against current active records (`is_current = true`) in the target table to find modified rows.
  3. **Perform Two-Part Merge:**
     * *Part 1:* For matching records with changes, update `is_current = false` and set `end_date` to the current date.
     * *Part 2:* For both new records and changed records, insert a new row with `is_current = true`, `effective_date = current_date`, and `end_date = null`.
  4. **Use Delta Lake MERGE:** Execute this logic in Spark using the Delta `MergeBuilder` API by joining target data with a unioned dataset of updates.
* **Common Mistakes:**
  * Forgetting to update the old record's `end_date` when inserting the new record (creates overlapping active intervals).
  * Implementing the logic using row-by-row loops in PySpark, which fails on large tables (set-based joins must be used).

## Scenario 8: Your Airflow DAG is consistently missing its SLA — what do you investigate and fix?
* **The Interviewer's Question:** *"An Airflow DAG that generates a daily business report is consistently missing its SLA of 7 AM. How would you investigate and resolve this?"*
* **What they are testing:** 
  * Airflow architecture debugging skills.
  * Task dependencies, resource allocation, and external system execution bottlenecks.
* **Structured Step-by-Step Approach:**
  1. **Identify the Bottleneck Task:** Open the Airflow UI, check the **Gantt Chart** and **Grid View** of the failed runs to find which specific task is taking the longest or waiting to start.
  2. **Check Scheduler & Queue Delays:** Look at task start times compared to scheduled start times. If tasks are stuck in `queued` status for a long time, the Airflow worker pool is exhausted. Fix this by adjusting worker counts or tuning concurrency parameters.
  3. **Inspect External Compute Engine Logs:** If the task is a `DatabricksSubmitRunOperator` or `SparkSubmitOperator`, the delay is occurring inside the Spark cluster. Open the Spark UI to check for data skew or job failures.
  4. **Check Sensors:** If the pipeline is waiting on an `S3KeySensor`, check if it's running in `reschedule` mode (good) or `poke` mode (bad, blocks worker slots).
  5. **Optimize Resources:** Allocate more resources to the worker nodes, scale out Celery workers, or optimize downstream queries.
* **Common Mistakes:**
  * Blaming Airflow when the bottleneck is actually the downstream query or Spark job.
  * Running sensors in `poke` mode for long-running checks, which exhausts worker slots and stalls all other DAGs.
  * Overlooking scheduler latency issues.

## Scenario 9: How do you design a data quality framework for a production pipeline?
* **The Interviewer's Question:** *"How would you design a data quality framework that validates our data before it reaches the Gold table, ensuring business users never see corrupt data?"*
* **What they are testing:** 
  * Understanding of Write-Audit-Publish (WAP) pattern.
  * Knowledge of data validation frameworks (e.g., Great Expectations, Soda, Deequ).
  * Error-routing strategies.
* **Structured Step-by-Step Approach:**
  1. **Implement Write-Audit-Publish (WAP) Pattern:**
     * *Write:* Write the output of the transformation step to a temporary staging table.
     * *Audit:* Run automated quality checks on the staging table (e.g., verify uniqueness, assert null constraints, check value ranges).
     * *Publish:* If checks pass, overwrite the production Gold table. If checks fail, halt execution and raise alerts.
  2. **Use Great Expectations / Soda:** Define declarative rules (e.g., `expect_column_values_to_not_be_null("user_id")`).
  3. **Establish Alerting:** Integrate test failures with Slack, PagerDuty, or Microsoft Teams.
  4. **Quarantine Strategy:** Route failing rows to a quarantine table so the pipeline can process valid rows instead of crashing completely.
* **Common Mistakes:**
  * Writing checks directly on production Gold tables *after* data is published (downstream dashboards consume corrupt data before check completes).
  * Failing the entire pipeline due to a single bad row (leads to operational overhead; routing to a quarantine table is better).

## Scenario 10: How do you make a pipeline fully idempotent?
* **The Interviewer's Question:** *"If we run your daily ETL pipeline three times for the same date, how do you guarantee that the target table contains no duplicate records and remains in a correct state?"*
* **What they are testing:** 
  * Concept of idempotency in big data storage.
  * Difference between Appending and Overwriting/Merging.
  * Handling temporal variables.
* **Structured Step-by-Step Approach:**
  1. **Avoid Append Mode:** Never use `.write.mode("append")` in Spark without a deduplication key or specific partition context.
  2. **Use Partition Overwrite:** If data is partitioned by date, use dynamic partition overwrite:
     ```python
     df.write.mode("overwrite").option("replaceWhere", "date = '2026-06-25'").save(path)
     ```
     This cleans and updates only the target date partition.
  3. **Use MERGE Statement:** For dimension tables, use a `MERGE` statement that updates existing records and inserts new ones based on a unique business key (e.g., `order_id`).
  4. **Log Offsets:** For streaming sources, commit consumer offsets only after completing the database transactions to ensure the pipeline resumes correctly on retry.
* **Common Mistakes:**
  * Using `UUID()` generation within transformation logic (produces new IDs on retry, creating duplicates).
  * Using simple appends without checking for duplicates.
  * Assuming that a full overwrite of a massive table is efficient (dynamic partition overwrite should be used instead).

## Scenario 11: You have a Spark job with a massive shuffle — how do you diagnose and fix it?
* **The Interviewer's Question:** *"Your Spark job is taking hours, and the Spark UI shows that a single stage is shuffling 5TB of data over the network. How do you diagnose and resolve this shuffle overhead?"*
* **What they are testing:** 
  * In-depth Spark execution mechanics.
  * Ability to minimize network shuffles.
  * Optimization of join and aggregation strategies.
* **Structured Step-by-Step Approach:**
  1. **Identify the Source of Shuffle:** Check the Spark UI **SQL Tab** and look for Exchange nodes (representing wide operations like `SortMergeJoin` or `HashAggregate`).
  2. **Reduce Source Data Size:** Apply filters and project only the columns needed (Projection Pushdown) before the join/aggregation step.
  3. **Broadcast Small Tables:** If one of the tables in a join is under 100MB, use `broadcast(df_small)` to convert a Sort-Merge Join into a Map-Side Broadcast Join, removing the shuffle.
  4. **Utilize Bucket-Map Joins:** If both tables are massive, pre-bucket both tables on the join column with the same number of buckets. Spark can perform a join on these files locally without shuffling data over the network.
  5. **Tune Concurrency:** Set `spark.sql.shuffle.partitions` to a value that balances partition sizes to around 128MB.
* **Common Mistakes:**
  * Running `.repartition()` multiple times (which triggers additional shuffles).
  * Joining on floating-point or unstructured text columns (creates inefficient hashing).

## Scenario 12: How would you design a pipeline that needs to support both historical backfill and daily incremental loads?
* **The Interviewer's Question:** *"We are deploying a new daily metrics table. How would you design the pipeline to process daily incremental loads efficiently, while also allowing us to backfill three years of historical data easily?"*
* **What they are testing:** 
  * Designing parameterizable pipelines.
  * Dynamic partitioning and resource scaling.
  * Handling backfills in Airflow.
* **Structured Step-by-Step Approach:**
  1. **Parameterize the Pipeline Date:** Design execution scripts (PySpark/SQL) to accept an input execution date parameter (e.g., `run_date`) rather than using hardcoded values like `current_date()`.
  2. **Partition Target Table by Date:** Store the target data partitioned by the event date (e.g., `PARTITIONED BY (event_date)`).
  3. **Use Dynamic Partition Overwrites:** Ensure Spark writes run in dynamic overwrite mode, replacing only the partition matching the input parameter:
     ```python
     df.write.mode("overwrite").option("partitionOverwriteMode", "dynamic").save(path)
     ```
  4. **Implement Airflow Backfill Execution:** In Airflow, set `catchup=True` on the DAG and specify a start date of three years ago. The scheduler will backfill the historical periods in parallel runs, controlled by `max_active_runs`.
* **Common Mistakes:**
  * Hardcoding dates in queries (e.g., `date = TODAY()`), which prevents historical backfilling.
  * Running a single massive Spark job to backfill three years of data at once (leads to OOM errors; processing in daily or monthly batches is more reliable).

## Scenario 13: How do you handle duplicate records arriving in a streaming pipeline?
* **The Interviewer's Question:** *"In our streaming transaction pipeline, network retries sometimes cause duplicate payment events to arrive. How do you deduplicate these events in real-time before writing them to the database?"*
* **What they are testing:** 
  * Stateful streaming deduplication.
  * Use of primary key merges and watermarking to manage state store growth.
* **Structured Step-by-Step Approach:**
  1. **Define Deduplication Key:** Identify the unique business identifier (e.g., `transaction_id`).
  2. **Apply Watermarking:** Specify a time threshold (e.g., `withWatermark("event_time", "1 hour")`) to instruct Spark to maintain a state store of processed IDs for up to 1 hour.
  3. **Use `dropDuplicates` in PySpark:** Call the drop duplicates function passing the key and event time:
     ```python
     df.withWatermark("event_time", "1 hour").dropDuplicates(["transaction_id", "event_time"])
     ```
  4. **Downstream Merge:** Write the stream to a Delta table using `foreachBatch` writer with a `MERGE` statement to handle duplicates that arrive outside the watermark window.
* **Common Mistakes:**
  * Running `dropDuplicates()` without a watermark (causes the streaming engine's state store to grow indefinitely, eventually causing an Out of Memory error).
  * Deduplicating based on processing time rather than event time.

## Scenario 14: Design the storage layer for a Data Lakehouse on Azure
* **The Interviewer's Question:** *"Describe how you would design the storage architecture for a Data Lakehouse on Azure. What tools, directory layouts, and security access controls would you use?"*
* **What they are testing:** 
  * Cloud infrastructure knowledge on Azure.
  * Standard storage directory structures (Medallion model).
  * Security policies (Access Control Lists, Service Principals, Unity Catalog).
* **Structured Step-by-Step Approach:**
  1. **Core Storage Account:** Use **Azure Data Lake Storage Gen2 (ADLS Gen2)** with Hierarchical Namespace enabled for file operations.
  2. **Directory / Container Structure:**
     * `abfss://raw@datalake.dfs.core.windows.net/` (Bronze layer, sub-directories grouped by `source_system/table_name/year/month/`)
     * `abfss://conformed@datalake.dfs.core.windows.net/` (Silver conformed Delta tables partitioned by business keys)
     * `abfss://enriched@datalake.dfs.core.windows.net/` (Gold aggregated reporting Delta tables)
  3. **Access Controls (IAM & ACLs):** Mount storage containers using **Microsoft Entra ID (Azure AD) Service Principals**. Apply fine-grained POSIX Access Control Lists (ACLs) on directories.
  4. **Governance Layer:** Enable **Unity Catalog** on Databricks to manage access control lists on tables, catalogs, and external locations using standard SQL `GRANT` queries.
* **Common Mistakes:**
  * Storing data in a flat container without a logical directory hierarchy.
  * Using account keys for mounting storage instead of Service Principals (creates a security vulnerability).
  * Granting all users read/write permissions directly to raw storage directories.

## Scenario 15: How do you decide between a managed table and an external table in Databricks?
* **The Interviewer's Question:** *"When building a Lakehouse in Databricks, when should we create a managed table, and when should we choose an external table? How does this impact our data governance?"*
* **What they are testing:** 
  * Understanding of Databricks storage architecture.
  * Risk management regarding data loss.
  * Integration with external query engines.
* **Structured Step-by-Step Approach:**
  1. **Define Managed Table Mechanics:** Managed tables are stored in the default workspace root warehouse location. If a user runs `DROP TABLE my_table`, Databricks deletes the metadata in the catalog and the physical Parquet files from cloud storage.
  2. **Define External Table Mechanics:** External tables are stored in a custom storage location (e.g., `s3://prod-data/users/`). If dropped, only the metadata entry in the catalog is removed; the physical files remain untouched.
  3. **Selection Criteria:**
     * *Choose Managed Tables:* For temporary staging tables, transient workspaces, or internal tables managed entirely by Unity Catalog.
     * *Choose External Tables:* For core production datasets, conformed tables shared with external analytical tools (e.g., PowerBI, Snowflake), and directories where data files must be protected from accidental deletion.
* **Common Mistakes:**
  * Choosing Managed Tables for production data without realizing that dropping the table will delete all historical files on disk.
  * Not configuring external tables with appropriate permissions, leading to permission drift between the storage level and catalog level.

## Scenario 16: Your Spark job is failing with an OutOfMemory (OOM) error on the Driver node
* **The Interviewer's Question:** *"Your daily analytics PySpark job keeps failing with a 'java.lang.OutOfMemoryError: Java heap space' error on the Driver node. How do you diagnose the cause and resolve this issue?"*
* **What they are testing:**
  * Architectural understanding of Driver vs Executor memory boundaries.
  * Identification of Spark operations that collect data to the master node.
  * Resolution strategies for handling broadcast table constraints and metadata overhead.
* **Structured Step-by-Step Approach:**
  1. **Locate the Trigger Action:** Review the job log stack trace to identify the line of code that triggered the crash. Look for actions that pull data from the distributed cluster back to the driver (e.g., `.collect()`, `.toPandas()`, `.show()`, or visualization calls).
  2. **Audit and Eliminate Large Collects:** If you find `.collect()` or `.toPandas()` on a large dataset, remove it. Perform computations (like filtering, grouping, or writing to disk) in a distributed manner using Spark DataFrame APIs. If you only need validation samples, replace it with `.take(10)` or `.limit(10)`.
  3. **Check Auto-Broadcast Join Settings:** If the OOM occurs during a join, verify if a large table was incorrectly broadcasted. The driver collects the broadcasted table first before broadcasting it to executors. If the table is larger than the driver memory, it will crash. Adjust `spark.sql.autoBroadcastJoinThreshold` to a lower value or remove broadcast hints on large tables.
  4. **Address Metadata Bloat:** If the table has millions of small files or partitions, the driver will OOM just by reading the metadata to build the file-scan plan. Run `OPTIMIZE` and `VACUUM` on the target Delta tables to compact small files.
  5. **Adjust Cluster Allocation:** If the DAG is highly complex with thousands of transformations (causing massive execution metadata), increase driver memory explicitly using `spark.driver.memory` in the Spark submit configuration.
* **Common Mistakes:**
  * Increasing executor memory (`spark.executor.memory`) instead of driver memory (which does not resolve driver-specific heap exhaustion).
  * Leaving debugging print statements (like `print(df.collect())`) in production code.
  * Assuming that calling `.toPandas()` on a Spark DataFrame is safe because "Pandas handles small datasets" (if the DataFrame scales in production, the driver will inevitably crash).

## Scenario 17: GDPR / CCPA "Right to be Forgotten" in a Data Lakehouse
* **The Interviewer's Question:** *"Our company operates a massive Data Lakehouse containing billions of customer records stored in Delta/Parquet files. Under GDPR/CCPA, when a user requests account deletion, how do we delete their personal data (PII) from our historical storage lake within 30 days without spending thousands in compute rewriting files?"*
* **What they are testing:**
  * Knowledge of table format operations (Delta/Iceberg DELETE, MERGE, and metadata rewriting).
  * Scale and cost-performance trade-offs in distributed data lakes.
  * Encryption/pseudonymization architectures for data erasure (Cryptographic Erasure).
* **Structured Step-by-Step Approach:**
  1. **Identify Physical Layout:** Store sensitive PII (names, emails) partitioned by a logical key or separate it from the main fact tables.
  2. **Approach 1: Cryptographic Erasure (Best Practice for Speed/Cost):**
     * Instead of physically deleting millions of rows from files (which requires rewriting complete files), encrypt every user's PII field with a unique customer encryption key stored in an external key manager (e.g. AWS KMS or Vault).
     * When a user requests deletion, destroy their specific encryption key from the key manager. The data remains in the lake but is now unreadable, cryptographically erased, costing near-zero compute.
  3. **Approach 2: Scheduled Delta / Iceberg DELETE (Compliance Batching):**
     * Accumulate deletion requests (e.g., in a `pending_deletions` table) and execute a single batch `DELETE FROM main_table WHERE user_id IN (SELECT user_id FROM pending_deletions)` once a week.
     * To clean up actual historical files and comply with GDPR physical deletion, run `VACUUM main_table RETAIN 0 HOURS` to purge historical Parquet files and remove transaction logs. (Make sure Spark config `spark.databricks.delta.vacuum.parallelDelete.enabled` is active to speed up physical deletes).
* **Common Mistakes:**
  * Executing a full Spark job to rewrite a 10TB table for every individual user deletion request (this is extremely expensive and causes write-conflict errors).
  * Forgetting to run `VACUUM` (without vacuuming, deleted data is still accessible using Delta/Iceberg Time Travel).
  * Storing raw PII in partition paths (e.g., `/user_name=john_doe/` because partition directories cannot be easily modified or deleted without table schema rebuilds).

## Scenario 18: Real-time Analytics Dashboard with High Concurrency & Sub-second Queries
* **The Interviewer's Question:** *"We need to design a customer-facing dashboard that displays payment stats in real-time. The system must ingest 50k events/sec from Kafka, support sub-second query latency, and handle 10,000 concurrent merchant queries. How would you design this?"*
* **What they are testing:**
  * Knowing that standard data lakehouses (Databricks/Spark) and classic warehouses (Snowflake) cannot handle high concurrency (thousands of queries/sec) or sub-second latencies efficiently due to query compilation overhead.
  * Selection of real-time OLAP engines (ClickHouse, Apache Druid, StarRocks, Pinot).
  * Stream processing vs database ingestion layouts.
* **Structured Step-by-Step Approach:**
  1. **Ingestion Layer:** Retain payment events in Kafka. Use Flink or Spark Streaming only if real-time transformations or cross-window joins are required. If raw ingestion is sufficient, use native database connectors.
  2. **OLAP Engine Selection:** Choose a real-time OLAP database like **ClickHouse**, **Apache Pinot**, or **Apache Druid** as the serving layer. These engines utilize columnar formats, vectorized execution, indexes, and aggregate projections to serve dashboard queries.
  3. **Handling Concurrency & Aggregations:**
     * Use **ClickHouse Materialized Views** with `SummingMergeTree` or `ReplacingMergeTree` engines to pre-calculate and aggregate metrics (e.g., transaction sums by merchant by minute) dynamically on ingest.
     * Downstream dashboard queries will read from these tiny pre-aggregated views instead of scanning billions of raw transactions.
  4. **Query Caching:** Place an in-memory caching layer (Redis) in front of the dashboard backend. Cache dashboard queries for 5-10 seconds to handle repetitive merchant refreshes.
* **Common Mistakes:**
  * Routing high-concurrency queries directly to a Data Lakehouse using Spark SQL or Athena (these engines are designed for batch throughput, not high concurrency, and will crash under 10k users).
  * Trying to write raw Kafka streams into Elasticsearch or standard MySQL at 50k events/sec (causes write bottlenecks and index-heap crashes).
  * Doing all calculations on read (e.g., `SELECT SUM(amount)...`) rather than utilizing pre-aggregated materialized views on write.

## Scenario 19: Design a payment audit trail pipeline with absolute integrity and zero data loss
* **The Interviewer's Question:** *"Design a payment audit trail pipeline. Every payment state change (INITIATED, SUCCESS, FAILED, REFUNDED) must be captured, archived, and queryable for audits. We require absolute data integrity, zero data loss, exact-once processing, and a tamper-proof audit log. How would you design this?"*
* **What they are testing:**
  - Ensuring zero data loss (relying on transactional write guarantees, WAL, Kafka configuration: `acks=all`, replication factor).
  - Exactly-once processing semantics (Kafka transactional API, Spark structured streaming with idempotent writes, or write-ahead logs).
  - Audit trail immutable design (append-only storage, digital signatures, hashing/chaining, WORM storage e.g. Amazon Glacier/S3 Object Lock).
  - Reconciliation mechanisms (detecting discrepancies between transactional state and audit logs).
* **Structured Step-by-Step Approach:**
  1. **Source / Producer Config:** Payment services publish transition events to a Kafka topic. Ensure the producer uses `acks=all`, `enable.idempotence=true`, and a replication factor of at least 3. Use schema validation (Avro/Protobuf) via Schema Registry to prevent structural regressions.
  2. **Streaming / Processing Layer:** Run a Spark Structured Streaming or Flink job with checkpointing enabled to an ACID storage (like S3/ADLS). Set up checkpointing with transactional target writes (e.g. Delta Lake/Iceberg).
  3. **Storage / Immutable Ledger Layer:**
     - Target storage is a Delta Table or Iceberg Table configured on a bucket with **S3 Object Lock (WORM - Write Once Read Many)** in Compliance mode. This prevents any deletion or modification of audit files, even by root/admin accounts.
     - Leverage Delta Lake's append-only constraint (`tblproperties('delta.appendOnly' = 'true')`) to enforce that only inserts are allowed.
  4. **Reconciliation / Audit Loop:** Run a daily batch job that reconciles the count and checksums of transactions in the operational Postgres DB with the records in the immutable audit lake. Flag any anomalies (e.g. missing transactions, state mismatches) to an Ops dashboard.
* **Common Mistakes:**
  - Using a standard relational database with updates to the same row for audit trails (defeats immutability; audit trails must be an append-only event stream).
  - Leaving Kafka producer config at defaults (`acks=1` or `acks=0`), which allows data loss if a broker crashes before replicating.
  - Allowing direct file deletion permissions on the cloud storage folder hosting the audit logs (fails compliance; requires WORM / Object Lock).

## Scenario 20: Design a cost-efficient backfill system for historical data at scale
* **The Interviewer's Question:** *"We have 3 years of raw clickstream logs in S3 (uncompressed, JSON format, ~500TB) and need to reprocess them through a new Spark pipeline to populate a new aggregated dimension table. Running this on a standard production cluster will blow up our monthly cloud budget. How would you design a cost-efficient, resilient backfill system?"*
* **What they are testing:**
  - Cost-optimization strategies (Spot instances vs On-demand, cloud storage tiering, data format optimizations).
  - Rate-limiting downstream writes (avoiding overloading target DWs/databases like RDS or DynamoDB).
  - Cluster configuration (Auto-scaling, Spark execution profiles, dynamic allocation).
  - Resiliency and progress tracking (checkpointing, avoiding re-running completed batches if a cluster fails mid-run).
* **Structured Step-by-Step Approach:**
  1. **Data Pre-flight Analysis & Tiering:** Check raw log structure. If archived, restore logs to a hot/cool tier. Process in parallel chunks rather than one giant window.
  2. **Compute Strategy (Spot Instances & Cluster Sizing):** Use transient clusters (e.g., AWS EMR or Databricks job clusters) configured with a small On-demand Master/Driver node (to prevent driver loss crashes) and 100% **Spot instances** for Core/Task executor nodes. This reduces compute costs by up to 90%.
  3. **State Management & Checkpointing (Idempotent Batches):** Use a control table (e.g., in DynamoDB or PostgreSQL) to track backfill progress by day/hour partitions. The pipeline checks the status table, picks up unprocessed partitions, and marks them as `RUNNING` then `SUCCESS`. If a Spot instance reclaim causes a worker node to drop and the job fails, the pipeline can resume from the last failed partition, preventing costly double-processing.
  4. **Dynamic Scaling & Serialization Options:** Use optimized formats like Parquet/ORC for reading if available; if raw JSON, enforce schema at read time to avoid expensive schema inference. Configure Spark to use `KryoSerializer` and tune partition size to around 128MB.
  5. **Downstream Rate Limiting:** If writing to an operational database (e.g. DynamoDB/RDS), implement rate-limiting or write-throttling in the Spark job to avoid triggering database capacity auto-scaling or knocking down online transaction services.
* **Common Mistakes:**
  - Running a single monolithic job for the entire 3 years (if it crashes at 95% complete, you have to restart from scratch, wasting thousands of dollars).
  - Using 100% Spot instances including the Driver node (if the Driver node gets reclaimed, the entire job terminates).
  - Using on-demand instances for heavy task nodes when Spot instances can do the exact same batch processing for a fraction of the cost.
  - Forgetting to throttle writes to the downstream database, which causes database CPU spikes and API throttling for the production app.
